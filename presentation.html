Legal Contract Q&A

Multi-Agent RAG System for Legal Contract Analysis


Console-based interactive system • Grounded answers with citations • Risk detection

1. Problem Statement

Build a system that lets users ask natural language questions about legal contracts and receive grounded, cited answers with risk indicators.

Requirements

  • Multi-turn conversational CLI
  • Answers backed by specific clauses
  • Legal/financial risk flagging
  • Cross-document conflict detection
  • Refuse out-of-scope requests

Corpus

  • NDA (Acme ↔ Vendor XYZ)
  • Vendor Services Agreement
  • Service Level Agreement
  • Data Processing Agreement

Topics: termination, confidentiality, liability, indemnification, governing law, data breach, compliance

2. System Architecture

2a. Ingestion Workflow

.txt files → Section Parser → Parent Chunks → Child Chunks (≤400 chars) → Embed (nomic-embed-text) → ChromaDB

Section-Aware Hierarchical Chunking

  • Parses document structure: numbered sections, Section N, Article N, ALL-CAPS headings
  • Parent chunks = full sections (rich context for LLM)
  • Child chunks = ≤400 char splits for precise retrieval
  • Sentence-aware splitting — never cuts mid-sentence
  • List-aware splitting — keeps (a), (b), (c) items together
  • 100-char overlap between consecutive children
  • Min 20 chars (set low to keep short but critical clauses like "99.5% uptime")

3. Retrieval Strategy

Three-Stage Hybrid Retrieval

  1. Vector search (ChromaDB + nomic-embed-text)
    Semantic similarity
  2. BM25 keyword search
    Exact term matches
  3. Score fusion
    70% vector + 30% BM25

+ Cross-Encoder Re-ranking

  • Top 20 candidates → ms-marco-MiniLM-L-6-v2
  • Joint (query, chunk) scoring → top 10

Why Hybrid?

  • Vector alone misses exact terms
    "Section 4.2" or "72 hours"
  • BM25 alone misses semantics
    "termination notice" ≈ "cancellation period"

HyDE (Hypothetical Document Embeddings)

  • Routing agent generates a hypothetical ideal clause
  • Used as vector query instead of raw question
  • Bridges question↔answer embedding gap

4. Evaluation — Limitations

  • Small query set — 17 queries cover key scenarios but not edge cases
  • No expert annotation — ground truths crafted by developer, not legal professionals
  • Local LLM judge — Ragas faithfulness unreliable with llama3.1 (timeouts, inconsistent decompositions); would improve significantly with GPT-4
  • Single-turn eval only — multi-turn conversation quality is not measured
  • No adversarial testing — prompt injection / jailbreak not evaluated

Custom deterministic metrics provide reproducible, fast regression checks — complement LLM-as-judge for complete coverage.

5. Known Pitfalls

PitfallImpactMitigation / Status
Regex-based chunkingNon-standard doc formatting → broken sectionsWorks well for structured contracts; would need layout-aware parsing for PDFs
Grounding verifier is an LLMCan mark hallucinated claims as "supported"CRAG catches most cases; no substitute for human review on critical queries
No confidence signalUser can't tell certain answers from guessesCould combine retrieval distance + grounding verdict into a score
Fragile JSON parsingLLM occasionally breaks JSON formatFallback parser with regex extraction; 95%+ success rate with llama3.1
Sequential 5-agent calls30-60s latency per query on CPURisk + consistency are independent → parallelizable for ~35% latency cut
BM25 rebuilt every startupSlow startup with large corpusOnly vector index is persisted; BM25 persistence would be a quick win
No prompt injection guardAdversarial queries could manipulate answersAdd lightweight classifier before routing agent

5. If This Were Production

P0 — Must Have

  • PDF/DOCX ingestion — layout-aware parsing; real legal docs are never plain text
  • Incremental indexing — re-embed only changed docs, not full rebuild
  • Confidence scores — combine retrieval distance + grounding result → user-facing indicator
  • Streaming — stream answer tokens while risk analysis runs in background

P1 — High Value

  • Observability — trace every agent call: input/output, tokens, latency, retrieval scores
  • Document versioning — audit trail: which contract version generated this answer?
  • Domain-adapted embeddings — fine-tune on legal text ("material breach" ≈ "significant violation")
  • User feedback loop — thumbs up/down → retrieval tuning + prompt refinement
